**外部サービス連携を前提にせず、それでも入力負担を最小化し、精度の高い成長判定を行うための完成形の設計**としてまとめます。

結論から言うと、レベログは単なる日報チャットではなく、次の3層で構成するのがよいです。

> **瞬間的に経験を残す「キャプチャ層」**
> **不足情報を効率的に集める「AIインタビュー層」**
> **複数の経験から能力を判断する「スキル評価層」**

AIに毎回長く質問させるのではなく、まず雑に記録を受け取り、必要な質問を適切なタイミングに分散させる設計が中心になります。

---

# 1. 解決すべき問題

外部連携を行わない場合、ユーザー自身による入力は避けられません。

したがって、目標は「入力をなくす」ではなく、次の状態を作ることです。

* 何を書けばよいか考えなくてよい
* 完成した文章を書かなくてよい
* 同じ説明を何度もしなくてよい
* その場で長い振り返りをしなくてよい
* AIの質問に答えるだけで記録が完成する
* 自分では気づかなかった成長が抽出される
* 入力した価値がすぐに返ってくる

ユーザーに日報を書かせるのではなく、**断片的な記憶をAIが成長データへ変換する**設計です。

---

# 2. 全体の体験設計

## 基本フロー

```text
ユーザーが雑に入力
        ↓
AIが経験内容を構造化
        ↓
不足情報と判定可能な項目を分析
        ↓
必要なら質問を1つ提示
        ↓
成長ログ案を生成
        ↓
ユーザーが確認・修正
        ↓
経験データとスキル評価を更新
        ↓
後日、類似経験と統合して再評価
```

重要なのは、入力直後にすべてを完成させようとしないことです。

ユーザーが忙しいときは一言だけ保存し、後でAIが質問します。

---

# 3. 入力体験を3つに分ける

## 3.1 クイックキャプチャ

最も軽い入力モードです。

ユーザーは文章を整えず、思ったことをそのまま入力します。

```text
API遅かったからSQL直した
```

```text
レビューで設計がおかしいと言われた
```

```text
今日の発表、説明はうまくいかなかった
```

```text
認証周りやっと理解できた
```

テキストだけでなく、次の入力方法を用意します。

* 音声入力
* ホーム画面のクイックボタン
* スマートフォンのウィジェット
* ブラウザやデスクトップのミニ入力画面
* 過去ログを複製して一部だけ変更
* 選択肢から開始
* 「できた」「困った」「学んだ」「決めた」の起点ボタン

入力後は、即座に長い質問を開始しません。

まずは保存して、

> 記録しました。SQL改善の経験として整理しています。

程度の反応だけ返します。

---

## 3.2 ガイド付き振り返り

時間がある場合は、その場でAIが深掘りします。

ユーザーは入力時に深さを選べます。

* 10秒で記録
* 1分で整理
* しっかり振り返る

「10秒で記録」なら即保存します。

「1分で整理」なら、重要な質問を1〜2個だけ聞きます。

「しっかり振り返る」なら、原因、判断、学び、再現性まで深掘りします。

AIが勝手に長い面接を始めるのではなく、**ユーザーがその瞬間に使える認知リソースに合わせる**設計です。

---

## 3.3 振り返りインボックス

情報不足のログは削除せず、「未整理の経験」として蓄積します。

```text
未整理の経験 3件

・APIのSQL改善
・レビューで設計を修正
・顧客向け説明会
```

週末や余裕のある時間に、AIが1件ずつ短い質問を出します。

> API改善について、結果は確認できましたか？

* 大きく改善した
* 少し改善した
* 変化はなかった
* 測定していない
* 覚えていない

この方式にすると、仕事直後に振り返りを強制せずに済みます。

---

# 4. AIインタビューの設計

AIは情報を集めるためではなく、**評価を変える情報だけを質問する**必要があります。

## 4.1 AIが抽出する基本情報

ユーザーの入力から、まず以下を抽出します。

### 経験の基本情報

* 何をしたか
* なぜ行ったか
* どのような状況だったか
* どのプロジェクトや領域か
* 誰に対する成果か

### 行動

* 調査
* 設計
* 実装
* 説明
* 判断
* 改善
* 支援
* 学習
* レビュー
* 合意形成

### 結果

* 完了した
* 一部完了した
* 解決できなかった
* 効果を確認した
* 効果は未確認
* 他者から評価された
* 次の課題が残った

### 能力判定に必要な情報

* 難易度
* 自走度
* 担当範囲
* 再現性
* 成果の客観性
* 他者の支援量
* 過去経験との関係
* 初めてか、繰り返し経験か

---

## 4.2 質問候補の生成

AIは不足しているすべての情報を聞くのではなく、複数の質問候補を作ります。

例：

```json
[
  {
    "question": "改善案は自分で考えましたか？",
    "target": "autonomy",
    "expected_information_gain": 0.89,
    "answer_effort": 0.15
  },
  {
    "question": "どの程度速度が改善しましたか？",
    "target": "outcome",
    "expected_information_gain": 0.76,
    "answer_effort": 0.32
  },
  {
    "question": "作業には何時間かかりましたか？",
    "target": "duration",
    "expected_information_gain": 0.18,
    "answer_effort": 0.23
  }
]
```

ここでは「自分で考えたか」が最優先です。

質問の優先度は、概念的には次のように計算します。

```text
質問価値
= 判定への影響度
× 回答による不確実性の減少
× 現時点で答えられる可能性
− 回答負担
− 既出情報との重複
```

質問数を固定するより、質問価値が一定以下になったら終了する方式が使いやすいです。

---

## 4.3 質問形式

基本は一度に一問です。

複数の質問を一画面に並べると、フォーム入力に戻ってしまいます。

質問には以下を併用します。

* 選択肢
* スライダー
* 数値入力
* 自由記述
* 「わからない」
* 「覚えていない」
* 「この質問は飛ばす」
* 「AIの推測で進める」

たとえば自走度なら、抽象的な5段階評価よりも、具体的な選択肢にします。

> どのように解決方法を見つけましたか？

* 自分で調査して見つけた
* 他の人と相談して見つけた
* ヒントをもらって自分で進めた
* 手順を教えてもらった
* 他の人が主に解決した

ユーザーに能力を自己採点させるより、**具体的な行動を選ばせてAIが評価に変換する**方が安定します。

---

# 5. 質問を途中で止める条件

次のいずれかを満たしたら、AIは質問を終了します。

* 成長ログを生成するための最低情報が揃った
* スキル判定の確信度が十分になった
* 次の質問による評価変化が小さい
* ユーザーの回答負担が高くなった
* ユーザーが後回しを選んだ
* 現時点では回答不能な情報しか残っていない
* 過去ログから補完できる
* 客観的な断定を避けるべき状態になった

質問終了後は、足りない情報を隠さず表示します。

```text
今回確認できたこと
・SQLのボトルネックを自分で特定した
・クエリを修正した

まだ確認できていないこと
・改善前後の具体的な数値
・類似問題を再現して解決できるか
```

---

# 6. 「できる・できない」の判定設計

単純な二値判定は避けます。

能力は次の状態で管理します。

```text
未経験
経験あり
支援を受けて実行可能
部分的に自立して実行可能
自立して実行可能
複雑な状況でも実行可能
他者へ説明・支援可能
設計や判断を主導可能
```

さらに、各能力について「何が確認できたか」を分けます。

## スキル状態の例

```json
{
  "skill": "SQL performance optimization",
  "state": "independent_basic_execution",
  "confidence": 0.74,
  "evidence_count": 3,
  "last_observed_at": "2026-07-20",
  "confirmed_capabilities": [
    "遅延原因の候補を調査できる",
    "基本的なクエリ改善を自立して行える"
  ],
  "unconfirmed_capabilities": [
    "大規模データでの実行計画分析",
    "インデックス戦略の設計",
    "継続的な監視設計"
  ]
}
```

「できない」と断定するのではなく、次の表現を使います。

* 未経験
* 判断材料がない
* 支援があれば実行できる
* 一部までは自立してできる
* 再現性が未確認
* 複雑な条件での実行は未確認
* 今回は完了できなかった
* 現在の課題として確認された

これなら、不足情報と能力不足を混同しません。

---

# 7. 経験と能力を分離する

重要なのは、1回の経験をそのまま恒久的な能力に変換しないことです。

## 経験ログ

そのとき何が起きたかを記録します。

```json
{
  "title": "低速APIのSQL改善",
  "status": "completed",
  "actions": [
    "原因調査",
    "実行クエリ確認",
    "SQL修正"
  ],
  "autonomy": "independent",
  "difficulty": "moderate",
  "outcome": "improved_unmeasured"
}
```

## スキルシグナル

その経験が、どの能力の証拠になるかを記録します。

```json
[
  {
    "skill": "SQL optimization",
    "signal": "positive",
    "strength": 0.72
  },
  {
    "skill": "problem diagnosis",
    "signal": "positive",
    "strength": 0.81
  },
  {
    "skill": "performance measurement",
    "signal": "unknown",
    "strength": 0
  }
]
```

## スキルプロフィール

複数のスキルシグナルを統合して、現在の能力状態を作ります。

```text
経験ログ
   ↓
スキルシグナル
   ↓
複数回の経験を統合
   ↓
現在のスキル状態
```

これにより、一度うまくいっただけで能力を過大評価したり、一度失敗しただけでレベルを下げたりすることを防げます。

---

# 8. 評価に使う軸

スキル評価では、少なくとも以下を別々に扱います。

## 経験量

その領域をどの程度経験しているか。

## 自走度

どの程度自分で判断して進められたか。

## 難易度

どれほど複雑な状況だったか。

## 成果

実際に目的を達成したか。

## 再現性

似た状況でも繰り返し実行できているか。

## 応用範囲

異なる条件やプロジェクトでも使えているか。

## 説明力

他者に説明、レビュー、支援できるか。

## メタ認知

自分の判断、失敗、改善点を説明できるか。

たとえば「React経験値100」だけではなく、

```text
React

実装経験          █████████░
自走度            ███████░░░
設計力            ██████░░░░
複雑な状態管理    ████░░░░░░
説明・レビュー    █████░░░░░
確信度            78%
```

のように、能力を多面的に見せます。

---

# 9. 過去ログを使った質問省略

入力負担を減らすうえで、最も重要な機能の一つです。

ユーザーが過去に、

* プロジェクトAではバックエンドを担当
* 使用技術はTypeScriptとPostgreSQL
* 基本的に一人で実装
* 上司が最終レビューを行う

と説明していれば、同じ情報を毎回質問しません。

新しい入力：

```text
また一覧APIを改善した
```

AIは過去の文脈から、

```text
プロジェクトAのPostgreSQLを使用した一覧API
```

と仮置きします。

ただし、勝手に確定せず、

> 前回と同じプロジェクトAの作業として整理しました。

という修正可能な表示にします。

AIの記憶は、次の3種類に分けます。

* ユーザーが明示的に設定した事実
* 複数回確認された安定情報
* AIが今回だけ仮定した情報

仮定した情報には必ず低い確信度を付けます。

# 10. AIの生成結果

質問終了後、AIは次の形式で結果を提示します。

## 生成例

```text
クエスト完了
低速APIのSQL改善

行ったこと
APIの遅延原因を調査し、SQLがボトルネックであることを特定。
クエリを修正し、レスポンスの改善を確認した。

今回確認できた能力
・問題の原因を切り分ける
・SQLのボトルネックを調査する
・基本的なクエリ改善を自立して行う

今回まだ確認できていないこと
・改善前後の定量的な測定
・大規模データでの最適化
・継続的な監視設計

スキルへの反映
SQL最適化        +14 EXP
問題解決          +11 EXP
バックエンド設計   +4 EXP

AI判定の確信度
74%
```

ユーザーは以下を操作できます。

* 確定
* 内容を修正
* スキルの関連付けを変更
* AIに再評価させる
* 質問に戻る
* この経験を評価対象外にする

# 13. AI処理はステートマシン化する

完全な自由会話として実装すると、質問がぶれたり、同じことを聞いたりしやすくなります。

内部では状態を明確に管理します。

```text
CAPTURED
   ↓
CLASSIFIED
   ↓
STRUCTURED
   ↓
GAP_ANALYZED
   ↓
QUESTIONING
   ↓
ASSESSABLE
   ↓
REVIEW_PENDING
   ↓
CONFIRMED
   ↓
SKILL_UPDATED
```

各ステートでAIができることを制限します。

たとえば `QUESTIONING` 状態では、成長ログを確定せず、質問と仮の抽出結果だけを生成します。

# 15. 証拠ベースの評価

AI判定には、必ず根拠を紐づけます。

```json
{
  "claim": "基本的なSQLボトルネックを自立して改善できる",
  "evidence": [
    {
      "type": "user_statement",
      "text": "自分で調査してSQLが原因だと特定した"
    },
    {
      "type": "activity_outcome",
      "text": "クエリ修正後に速度が改善した"
    }
  ],
  "confidence": 0.74
}
```

画面上でも、「なぜこの判定になったか」を確認できるようにします。

AIのブラックボックス感を減らすことで、スキル評価への納得感が上がります。

# 16. ユーザー修正を学習データとして扱う

ユーザーが、

```text
AI：設計力の経験として判定
ユーザー：設計ではなく実装作業だった
```

と修正した場合、その変更を記録します。

```json
{
  "original_assessment": "architecture_design",
  "corrected_assessment": "implementation",
  "correction_reason": "user_override"
}
```

以後、同じユーザーについては判断傾向を調整します。

ただし、モデル自体をその場で学習させるのではなく、ユーザー固有の評価ルールやコンテキストとして利用します。

# 17. 失敗や未完了の扱い

未完了の経験も、通常のログとして保存します。

```text
クエスト継続中
認証エラーの原因調査

今回できたこと
・再現条件を特定した
・フロントエンド側ではないことを確認した
・トークン検証処理まで原因を絞った

まだできていないこと
・根本原因の特定
・修正
・再発防止

次に試すこと
・期限切れトークンの検証
・サーバー時刻の確認
```

この場合、問題解決の完成経験値は与えなくても、調査、切り分け、仮説構築の経験として記録できます。

また、未完了クエストとして後から更新できます。

# 18. 入力負担を増やさないためのルール

実装上、次のルールを設けるべきです。

* 同じことを二度聞かない
* 一度に一問だけ聞く
* 回答できない選択肢を必ず用意する
* 質問を後回しにできる
* 最初の入力は一文でよい
* AIが推測した情報は修正可能にする
* 数値がなくても記録できる
* 詳細を書かなくても最低限の価値を返す
* 長文入力を要求しない
* 毎日入力しなくても不利益を与えない
* 連続記録を過度に評価しない
* 空白期間を失敗として扱わない

ストリークを中心にすると、入力できなかった罪悪感がアプリ離脱を加速させます。

レベログでは、継続日数より**蓄積された経験の質と幅**を中心にする方が適しています。

# 19. 通知設計

通知は「記録してください」ではなく、回答しやすい具体的な問いにします。

悪い通知：

```text
今日の振り返りを入力しましょう
```

良い通知：

```text
昨日のAPI改善は、結果まで確認できましたか？
```

```text
「認証エラーの調査」に進展があれば、一言だけ残せます。
```

```text
今週は問題解決の経験が3件ありました。
AIが共通点を整理しました。
```

通知から直接選択肢に回答できると、さらに負担を下げられます。

# 20. 最終的な設計方針

この懸念に対する最も使いやすい実装は、次の形です。

## ユーザーの責任

* 起きたことを一言で残す
* AIの質問に、答えられる範囲で答える
* 最終結果を確認する

## AIの責任

* 雑な入力を理解する
* 過去の文脈を利用する
* 不足情報を特定する
* 判定を変える質問だけを聞く
* 質問を適切なタイミングに分散する
* 経験を構造化する
* 事実と推測を分ける
* 能力の根拠と未確認部分を示す
* 複数の経験から長期的な能力を判断する

## システムの責任

* AIの出力を検証する
* 変更履歴を保存する
* 評価の根拠を保持する
* ユーザーによる修正を優先する
* 個別の経験とスキル状態を分離する
* 不確実な情報を断定しない

この設計における中心体験は、

> **一言だけ残した経験が、AIとの短い対話を通じて、自分の能力と課題として整理される**

ことです。

「毎日振り返りを書くアプリ」ではなく、**経験の断片を預けると、AIが時間をかけて自分の成長地図を組み立ててくれるアプリ**として設計するのが最も自然です。
