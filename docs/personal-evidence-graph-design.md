# gpt output
あとはやってきたこととか、めちゃくちゃ取り込めるような感じにした方がいいですね。いろんなメモとか、例えば僕だったらObsidianにめちゃくちゃ情報があるので、それをまあ構造化してため込めるようにしておいた方がいいですね。
例えば、今だったらプロフィールでいろいろ自分のパーソナリティを入れられるようになっているのですが、これで過去に取っておいたメモとかをどんどん追加しておいて、テキストだったりマークダウンファイルだったりとかで追加して、どんどん自分のパーソナリティをめちゃくちゃパーソナライズできるようにした方がいいですね。


それ、レベログの価値をかなり大きくできます。

単なる「日々の成長記録」ではなく、

> **過去のメモ、経験、成果物、思考を取り込み、本人の能力・実績・価値観を再構成する個人データ基盤**

にできます。

ただし、重要なのは、取り込んだ情報をすべて「パーソナリティ」として扱わないことです。
パーソナリティ、知識、経験、実績、興味関心は、それぞれ別の層として保存した方がよいです。

# 全体構造

```text
Obsidian・Markdown・テキスト・過去ログ
                ↓
        インポート・解析
                ↓
      個人エビデンスデータベース
                ↓
 ┌────────┬────────┬────────┐
 │プロフィール│スキル評価│経験・実績│知識・関心│
 └────────┴────────┴────────┘
                ↓
 ポートフォリオ・職務経歴書・自己紹介・成長分析
```

レベログの中心に、ユーザー本人に関する**Personal Evidence Graph**のようなものを置くイメージです。

日本語なら、

* 個人エビデンス基盤
* 経験グラフ
* 成長ナレッジベース
* パーソナルアーカイブ

あたりの概念になります。

# 取り込むデータを分ける

## 1. 明示プロフィール

本人が直接設定した情報です。

* 職種
* 経歴
* 得意分野
* 目標
* 価値観
* 興味
* 働き方
* 好きな技術
* 将来やりたいこと

これは本人が明示した情報なので、比較的強い事実として扱えます。

## 2. 経験・実績

本人が実際に行ったことです。

* プロジェクト
* 開発経験
* 問題解決
* 意思決定
* チームへの貢献
* 登壇
* 記事執筆
* 学習
* 失敗
* 改善
* 成果物

ポートフォリオの中心になるデータです。

## 3. スキルエビデンス

経験から抽出された能力の根拠です。

```text
「Cloudflare WorkersでAPIを実装した」
        ↓
Cloudflare Workersの実装経験
サーバーレス設計
TypeScript
API設計
デプロイ経験
```

単にスキル名を登録するのではなく、**どの記録がそのスキルの根拠なのか**を保持します。

## 4. 知識・思考

Obsidianに大量にありそうな部分です。

* 技術メモ
* 読書メモ
* 設計思想
* アイデア
* 仮説
* 調査結果
* 学習ノート
* 日記
* 振り返り
* 意思決定の理由

これらは、直ちに「できること」には変換しません。

たとえばReactについて詳しいメモがあっても、実際にReactで開発した証拠とは限らないからです。

そのため、

```text
知っている可能性がある
理解を深めている
実際に使用した
成果を出した
他者に説明できる
```

を分けて評価する必要があります。

## 5. パーソナリティ・傾向

複数の記録から推定される傾向です。

* 新しい技術を試す傾向
* 問題を構造化して考える
* 納得できるまで調査する
* UIへの関心が強い
* 自動化を好む
* 一人で考えてから相談する
* 品質を優先する

ただし、これはあくまでAIによる推定です。

```text
本人が明示した性格
AIが記録から推定した傾向
一時的な状態
```

を混ぜずに表示するべきです。

# インポート機能

ユーザーが過去の情報を一気に投入できる、**インポートインボックス**を用意します。

対応対象としては、次のようなものが考えられます。

* Markdownファイル
* テキストファイル
* Obsidian Vault
* ZIPファイル
* PDF
* Word
* JSON
* CSV
* コピー＆ペースト
* 複数ファイルのドラッグ＆ドロップ
* Webページの保存データ
* 過去の職務経歴書
* 過去のポートフォリオ
* ブログ記事
* 日報
* 議事録
* アイデアメモ

Obsidianの場合は、Vault全体をZIPでアップロードできる形が扱いやすいです。

```text
my-vault.zip
├── Projects/
├── Daily Notes/
├── Books/
├── Ideas/
├── Development/
└── Career/
```

フォルダ構成、タグ、内部リンク、frontmatterも解析対象にします。

# インポート処理

## 1. 原文を保存する

まず元ファイルをそのまま保存します。

AIが抽出した情報だけを残すと、後から解析方法を改善した際に再処理できません。

```json
{
  "source_type": "obsidian_markdown",
  "source_path": "Projects/levelog.md",
  "raw_content": "...",
  "imported_at": "2026-07-20"
}
```

## 2. 文書を分類する

AIが文書の種類を判定します。

* プロジェクト記録
* 技術メモ
* 日記
* 振り返り
* アイデア
* 読書メモ
* 実績
* タスク
* 議事録
* キャリア情報

複数カテゴリを持たせてもよいです。

## 3. 情報を抽出する

文書から構造化データを抽出します。

```json
{
  "projects": [],
  "activities": [],
  "skills": [],
  "decisions": [],
  "outcomes": [],
  "lessons": [],
  "people": [],
  "organizations": [],
  "technologies": [],
  "interests": [],
  "personality_signals": []
}
```

## 4. 既存データと統合する

同じプロジェクトや経験が複数のメモに登場する場合、別々の実績として作らないようにします。

たとえば、

```text
levelog.md
levelog-ui.md
levelog-db-design.md
2026-07-20-daily.md
```

を、同一の「レベログ開発プロジェクト」に関連付けます。

## 5. AIが不明点を確認する

AIが勝手に確定せず、重要な曖昧さだけを質問します。

> 「Levelog」と「レベログ」は同じプロジェクトですか？

> この設計メモは、実際に実装した内容ですか？
> それとも検討段階のアイデアですか？

> このプロジェクトは個人開発ですか、業務プロジェクトですか？

この確認によって、構想と実績を混同する事故を防げます。

# 取り込み後の画面

大量のファイルを入れた直後に、全部確認させるのはしんどいです。

そのため、まずAIが概要を表示します。

```text
IMPORT ANALYSIS COMPLETE

245件のノートを解析しました

検出された内容
・プロジェクト候補      18件
・技術的な経験         126件
・意思決定             43件
・学習記録             89件
・成果候補             27件
・未確認の人物・組織    12件

特に多く記録されている領域
・Webアプリケーション開発
・AI画像・動画生成
・UI設計
・Cloudflare

```

その後、AIが価値の高い確認から順に出します。

> 2025年に記録された「動画生成ツール開発」は、実際に公開しましたか？

この答えによって、単なるアイデアから公開実績へ昇格します。

# ポートフォリオの公開設計

個人データを扱うため、公開範囲は細かく制御できる必要があります。

```text
非公開
本人だけが閲覧

限定公開
URLを知っている人だけ閲覧

項目単位で公開
技術・実績のみ公開
日記・内部メモは非公開

公開ポートフォリオ
外部向けに編集済みの情報だけ公開
```

原文のメモをそのまま公開するのではなく、公開用のビューを別に生成します。

```text
原文
「認証マジで意味わからん。3時間溶けた」

公開用
「認証フローの不整合を調査し、トークン検証処理を改善」
```

ここはAIがいい具合に社会性のシャツを着せてくれます。

# 「事実」と「AIの解釈」を分ける

データには出所を付けます。

```json
{
  "claim": "UI設計を得意としている",
  "source_type": "ai_inference",
  "confidence": 0.81,
  "evidence": [
    "UI設計に関するノートが42件ある",
    "3つのプロジェクトでUI設計を担当",
    "本人がUI設計への関心を明示"
  ]
}
```

表示上も、

* 本人が登録
* インポートした記録から抽出
* AIが推定
* 複数の実績から確認
* 未確認

といったラベルを付けます。

# パーソナライズへの利用

蓄積データは、単にプロフィール表示に使うだけではありません。

AIの応答自体をユーザーに合わせられます。

たとえばレベログのAIが理解している状態です。

```text
このユーザーは
・フルスタック開発の経験がある
・抽象的な説明より実装案を好む
・Cloudflareを頻繁に使う
・UIへの関心が強い
・個人開発を継続している
・過去に認証設計で苦戦した
```

すると新しい相談に対して、

> 前回の認証設計ではトークン更新処理が課題になっていたので、今回は先にセッション管理を分離するとよさそうです。

のように、過去の経験を踏まえた回答ができます。

ただし、AIに渡す情報は毎回全件ではなく、相談内容に関連するデータだけを検索して渡します。

# 重要な設計原則

この機能では、次の原則がかなり重要です。

1. **原文を消さない**
   AIの解析結果だけでなく、元データを保持する。

2. **構想と実績を区別する**
   アイデアを書いたことと、実装したことは別。

3. **知識と能力を区別する**
   メモがあることと、実務で使えることは別。

4. **本人の明示情報とAI推定を区別する**
   AIの解釈を事実として扱わない。

5. **すべてを確認させない**
   重要度が高い曖昧さだけを質問する。

6. **いつでも修正できる**
   AIの分類、スキル、プロジェクト統合をユーザーが直せる。

7. **公開データを分離する**
   個人アーカイブと外部向けポートフォリオを別レイヤーにする。

# レベログのプロダクト定義

この方向まで広げると、レベログは、

> 日々の経験を記録するアプリ

というより、

> **人生の中で生まれた経験、知識、成果、思考を蓄積し、自分が何者で、何ができ、何を目指しているのかを更新し続けるパーソナルデータベース**

になります。

そしてポートフォリオは別途頑張って書くものではなく、**普段ためているデータから必要な瞬間に生成される出力形式の一つ**になります。

このテーマは既存の `growth-log-system-design.md` とは分けて、次の設計書として持つのがよさそうです。

```text
personal-knowledge-ingestion-design.md
```

または、プロダクトの中核概念として広く扱うなら、

```text
personal-evidence-graph-design.md
```

がかなりしっくりきます。
