# Levelog MVP Implementation Plan

Last updated: 2026-07-20 (Asia/Tokyo)

This document is the source of truth for implementation and session handoff. Only the orchestrator updates task status and integration notes. Agents report results to the orchestrator and do not edit this file.

## Product objective

Levelog is a local macOS growth OS that turns daily work and learning into user-confirmed growth evidence and a safe next challenge. The product must not treat AI output as fact or convert activity volume directly into skill mastery.

The MVP is complete when a user can:

1. Configure a role, one to three focus skills, weekly time budget, excluded quest patterns, and a Codex CLI path.
2. Save an activity using any of: what they did, what was difficult, or what changed.
3. Review and optionally redact the exact payload before it is sent to Codex.
4. Review, edit, accept, or reject AI analysis candidates.
5. Confirm analysis so accepted evidence and deterministic account XP are recorded atomically.
6. Receive, accept, adjust, postpone, cancel, complete, and reflect on a safe quest.
7. See real activity, XP, quest, and evidence data in the mock-inspired dashboard.
8. Create a safe SQLite backup and a versioned JSON export.

## Non-goals

- App Store distribution, signing, or notarization
- Windows or mobile support
- External integrations such as Obsidian, GitHub, Calendar, Slack, or cloud sync
- Social, ranking, guild, or organization features
- Weekly review, calibration quests, automatic skill rank promotion, semantic search, or a vector database
- Fully local inference; Codex with ChatGPT login may use cloud inference
- Functional search and notification systems in the first release

## Design system

- Base background: deep navy, never pure black.
- Primary system color: cyan blue; success/growth: green; primary action/reward: amber-gold; special proposals: purple.
- Panels use thin visible borders and clipped corners. Blur is limited; hierarchy comes from borders, gradients, and restrained inner glow.
- Typography uses the macOS system Japanese stack and a system monospace stack for numbers. No runtime font download.
- All controls use semantic elements, visible focus, minimum 44x44 hit areas, labels, and non-color state indicators.
- Motion uses only transform/opacity, 150-300 ms, and respects `prefers-reduced-motion`.
- Charts and the growth core use CSS/SVG. No R3F or chart dependency.
- Responsive layout:
  - `>=1360px`: three-column mock-style dashboard
  - `1100-1359px`: two-column dashboard
  - `1024-1099px`: icon sidebar and vertically stacked panels
- Initial Tauri window: 1440x900; minimum: 1024x720.

## Fixed domain rules

### Skills

The internal skill catalog is fixed to 15 IDs across five categories:

```text
thinking.information_structuring
thinking.problem_decomposition
thinking.hypothesis_testing
technical.technical_learning
technical.system_design
technical.validation
communication.clarification
communication.explanation
communication.documentation
execution.prioritization
execution.planning
execution.follow_through
interpersonal.listening
interpersonal.alignment
interpersonal.feedback
```

AI can only reference these IDs. The first MVP displays evidence count and `observing`, not a synthetic mastery rank.

### XP and levels

- Activity saved: `+10`, once per activity.
- Analysis confirmed: `+20`, once per analysis/activity confirmation.
- Quest reflection saved: `+40`, once per quest, regardless of outcome.
- Candidate count, accepted evidence count, and quest success do not grant XP.
- Cumulative XP required to reach level L: `50 * (L - 1) * L`.
- XP is an append-only ledger and every award has a unique reason key.

### Quest state

Allowed states:

```text
proposed -> accepted -> in_progress -> completed
proposed|accepted|in_progress -> rescheduled
proposed|accepted|in_progress -> adjusted
proposed|accepted|in_progress -> cancelled
rescheduled|adjusted -> accepted|in_progress|cancelled
```

Rescheduling and adjustment are not failures. A reflection can record `completed`, `partially_completed`, `not_completed`, or `rested`; each is equally eligible for the reflection XP event.

Quest generation uses a versioned, safe template catalog. A generated quest must be within the user's authority, short, observable, reversible, and avoid unnecessary burden on other people.

## Persistence contract

The initial migration creates:

- `activities`
- `skills`
- `ai_analyses`
- `skill_candidates`
- `skill_observations`
- `quests`
- `quest_reflections`
- `xp_events`
- `app_settings`

Invariants:

- IDs are UUID strings; timestamps are UTC ISO 8601; activity occurrence also stores a local calendar date.
- `PRAGMA foreign_keys = ON`, WAL, and 5000 ms busy timeout are enabled on every pool.
- User-entered activity fields are immutable after AI analysis begins. Corrections create an explicit new revision or new activity; AI never overwrites raw text.
- Submitted/redacted payload, raw AI result, provider, model, Codex version, prompt version, and schema version are retained with the analysis.
- AI candidates remain separate from user decisions and confirmed observations.
- Analysis confirmation and its XP/evidence writes are one transaction and idempotent.
- Migrations are append-only after first release.

Backup and export:

- Database: Tauri's Application Support app-data directory (`com.levelog.desktop/levelog.db`).
- On startup, at most once per local day, create a consistent backup with SQLite `VACUUM INTO`.
- Manual backup uses the same mechanism.
- JSON export includes schema version, export timestamp, settings excluding future secrets, raw activities, analysis/user decisions, observations, quests/reflections, and XP events.
- Restore/import UI is not part of the first MVP.

## Tauri command and DTO contract

The frontend can invoke only purpose-specific commands; it never receives raw SQL or arbitrary process execution.

```text
get_boot_state() -> BootState
save_onboarding(OnboardingInput) -> AppSettingsDto
update_codex_path(CodexConnectionInput) -> AppSettingsDto
test_codex_connection(CodexConnectionInput) -> CodexConnectionStatus

get_dashboard() -> DashboardSnapshot
list_skills() -> SkillDto[]
create_activity(CreateActivityInput) -> ActivityDto
list_activities() -> ActivityDto[]
get_activity(activityId) -> ActivityDetailDto

get_analysis_preview(activityId) -> AnalysisPreview
start_activity_analysis(StartAnalysisInput) -> AnalysisJobDto
get_activity_analysis(analysisId) -> ActivityAnalysisDto
cancel_activity_analysis(analysisId) -> AnalysisJobDto
confirm_activity_analysis(ConfirmAnalysisInput) -> ConfirmAnalysisResult

generate_quest(GenerateQuestInput) -> QuestDto
list_quests() -> QuestDto[]
transition_quest(QuestTransitionInput) -> QuestDto
save_quest_reflection(QuestReflectionInput) -> QuestReflectionDto

create_backup() -> BackupResult
export_json() -> ExportResult
```

`AnalysisPreview` contains the exact JSON to be submitted and an editable `submittedPayload`. Editing it does not edit the source activity.

`ActivityAnalysis` schema v1 contains:

```text
summary: string
outcomes: string[]
skillCandidates: [{ skillId, confidence 0..1, reason, evidence }]
missingInformationQuestion: string|null
```

`QuestProposal` schema v1 contains:

```text
templateId: string
title: string
description: string
targetSkillId: string
estimatedMinutes: integer
difficulty: integer 1..5
successCriteria: string[]
evidencePrompt: string
```

## Codex execution boundary

- Save the activity before starting inference.
- Resolve and persist an absolute Codex path; verify `codex --version` and `codex login status`.
- Execute directly with `tokio::process::Command`, never through a shell.
- Use a new empty temporary working directory for every call.
- Required controls: ephemeral run, read-only sandbox, ignore user config, ignore rules, skip Git check, disable Web search, disable shell/unified exec/browser/computer tools, output schema, and output-last-message.
- If the installed CLI does not support a required safety control, return a compatibility error. Never silently remove the control.
- Only one inference job runs at a time. Timeout is 180 seconds. It is cancellable.
- Retry once only when output is missing or violates the schema.
- On startup, stale `running` jobs become `failed` with an interrupted reason and can be retried.
- Runtime has no mock inference provider. Tests inject a fake `ProcessRunner`.

## Resume procedure

When resuming after an interrupted session:

1. Read this file completely.
2. Run `git status --short --branch` and `rg --files`.
3. Find the first `in_progress` task in the ledger. If none exists, read `Known issues and deliberate limits` and `Next task` before starting new scope.
4. Run the most local test for that task, then continue it; do not restart completed work.
5. Before dispatching an agent, confirm its ownership does not overlap another live agent.
6. After integration, update the ledger, completed work, verification results, known issues, and `Next task`.
